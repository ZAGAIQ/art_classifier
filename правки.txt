Отчёт о соответствии исходного кода и научного отчёта ОПР («АртВизор»)
Настоящий отчёт содержит детальный анализ соответствия между текстовым документом 
АртВизор_Отчёт_ОПР.docx
 (далее — Отчёт ОПР) и фактической программной реализацией в репозитории проекта (бэкенд 
app.py
, фронтенд 
templates/index.html
, скрипты обучения и метрики).

Сравнение выявило критические расхождения, которые приведут к ошибкам при проверке интеграции API, а также существенное занижение научных результатов в тексте работы.

1. Критические нестыковки в API и коде
Данные расхождения критичны, так как спецификация API в Отчёте ОПР полностью расходится с тем, что возвращает бэкенд на самом деле. Любой автоматический или ручной тест по спецификации Отчёта завершится ошибкой.

1.1. Формат ответа эндпоинта анализа
В Отчёте ОПР (Раздел 6.1):
Путь: POST /analyze
Поле probability: вероятность в процентах (шкала 0..100).
Поле is_impressionist (boolean: true/false).
Поле threshold: порог классификации (50.0%).
Массив похожих работ: similar[] с полями artist, title, similarity (%), filename.
В коде (
app.py:L105-166
):
Путь: POST /api/analyze (с префиксом /api).
Поле probability: возвращается как float (шкала 0..1), например 0.8984....
Поле is_impressionism (а не is_impressionist).
Поле threshold в ответе отсутствует полностью.
Массив похожих работ: similar_images[] (а не similar).
Влияние на работу: Фронтенд 
templates/index.html
 написан под реальный код бэкенда (использует /api/analyze, is_impressionism и similar_images). Однако, если оценивать систему по Отчёту ОПР, спецификация API не совпадает с кодом.
1.2. Имя файла фронтенда
В Отчёте ОПР (Разделы 1.2, 2.2, 6.2): Утверждается, что интерфейс реализован в файле artvizor_mvp.html, который пользователь запускает в браузере.
В коде: Фронтенд находится в файле 
templates/index.html
. Файла artvizor_mvp.html в репозитории нет.
1.3. Ложное утверждение о CORS (BUG-007)
В Отчёте ОПР (Разделы 6.2, 7.3): Утверждается, что дефект BUG-007 (CORS-ошибка) закрыт, а библиотека Flask-CORS импортирована и настроена на origins=['*'].
В коде (
app.py
): Нет импорта flask_cors и настройки CORS. В файле 
requirements.txt
 библиотека flask-cors отсутствует. При попытке сделать запрос с другого источника (например, localhost:8080 или локального файла) возникнет CORS-блокировка.
1.4. Проверка размера файлов и компрессия
В Отчёте ОПР (Разделы 3.3, 6.2, 7.3): Заявлено, что для файлов > 8 МБ на бэкенде настроена валидация с отдачей кода 413 Payload Too Large, а также Pillow-сжатие, а на фронтенде — клиентская компрессия через Canvas API для файлов > 5 МБ.
В коде: Никаких проверок размера и Pillow-компрессии в 
app.py
 нет. Фронтенд 
templates/index.html
 также отправляет исходный файл напрямую через FormData без Canvas-компрессии.
2. Занижение научных результатов в тексте работы
Данные пункты критически важны для повышения научного статуса работы. Текст Отчёта ОПР существенно преуменьшает масштабы проделанной вами работы.

2.1. Объём датасета (в 25 раз больше!)
В Отчёте ОПР (Разделы 1.2, 4.1, 8.1): Заявлено, что собран датасет из «200+ изображений».
В коде и реальности: Фактический размер каталога 
balanced_art_dataset
 составляет 4921 изображение (почти 5000!).
Рекомендация: Обязательно исправьте в Отчёте ОПР «200+ изображений» на «около 5000 сбалансированных изображений». Для научной работы датасет объёмом 5000 единиц выглядит намного весомее и репрезентативнее, чем скромные 200+.
2.2. Качество модели (Точность на 10% выше!)
В Отчёте ОПР (Разделы 8.1, 8.2): Заявлена итоговая точность (Accuracy) классификатора 80% (цель $\ge$ 70%).
В коде и логах (
metrics.json
): Реальная валидационная точность модели составляет 89.85% (почти 90%), а обучающая — 90.52%.
Рекомендация: Поменяйте в выводах 80% на 90%. Точность в 90% для задачи классификации стилей живописи — это отличный результат научной работы.
2.3. Размер тестовой/валидационной выборки
В Отчёте ОПР (Раздел 5.1): Анализ проводится на «тестовой выборке из 25 изображений».
В коде и логах (
metrics.json
): Матрица ошибок и метрики в проекте рассчитаны на основе валидационной выборки (20% от датасета), содержащей 985 изображений (437 TN, 62 FP, 38 FN, 448 TP).
Рекомендация: Укажите в отчёте, что оценка модели производилась на репрезентативной валидационной выборке из 985 изображений, а не на 25 картинках. Это в десятки раз увеличивает статистическую достоверность ваших выводов.
3. Функциональные и структурные расхождения
3.1. Ленивая загрузка моделей (Lazy Loading)
В Отчёте ОПР (Раздел 3.3, 6.1): Написано, что для избежания долгого старта сервера применена ленивая загрузка моделей (lazy loading) при первом запросе.
В коде (
app.py:L167-169
): Функция init_app() вызывается синхронно при запуске сервера в блоке __main__. То есть модели загружаются сразу при старте скрипта, увеличивая время запуска сервера, а не при первом API-запросе.
3.2. Количество рекомендуемых картин (Top N)
В Отчёте ОПР (Разделы 2.2, 6.1, 8.1): Заявлен вывод Топ-5 похожих работ.
В коде (
app.py:L115
): Значение по умолчанию для top_n в бэкенде установлено равным 6 (top_n = int(request.form.get('top_n', 6))).
3.3. Название проекта
В Отчёте ОПР: Проект называется «АртВизор» (ArtVizor).
В коде и README: В 
README.md
 проект назван «Классификатор произведений импрессионизма (Art Classifier)», а в веб-интерфейсе 
templates/index.html
 — «Impressionism AI» / «Impressionism Classifier Prototype».
4. Что совпадает идеально
Архитектура признаков: Модель openai/clip-vit-base-patch32 используется во всех скриптах для извлечения 512-мерных векторов.
Классификатор: Однослойная Keras-модель с L2-регуляризацией (Dense(1, activation="sigmoid")), имеющая ровно 513 параметров.
Список стилей датасета: Стили в датасете соответствуют перечисленным в отчёте: Impressionism, Post_Impressionism, Baroque, Cubism, Minimalism, Ukiyo_e.
Математика рекомендаций: Косинусное сходство через скалярное произведение L2-нормализованных векторов (np.dot(imp_embeddings, query_emb)).
Консольный скрипт: Логика работы 
analyze_image.py
 и его аргументы командной строки полностью совпадают с описанными в Отчёте ОПР.
